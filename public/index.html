平台定位

统一大模型接入层，帮助团队把“先接上、再治理、后扩展”放到同一平台完成。

适用对象

个人开发者、AI 产品团队、内部共享网关与多客户端接入场景均可直接使用。

官方口径

能力、价格、权益和限制均以控制台配置、接口文档与实际账户状态为准。

为什么先接入平台层 operational ready

多数团队真正需要统一的不是“某一个模型”，而是接入方式、调用策略、计费口径和运行期控制。平台层的价值在于先缩短接入周期，再降低后续迁移、排障与运营成本。

协议差异在平台层消化

对接 SDK、CLI 与第三方客户端时，减少为不同上游分别维护接入逻辑的成本。

模型切换从硬编码变为运行期选择

模型、路由、价格和可用性在控制台与平台层统一呈现，业务入口无需频繁重构。

计费与日志统一归档

调用日志、额度消耗、套餐权益和异常状态可以在同一控制面查看，便于快速定位问题。

统一 API Key 统一控制台 统一路由策略 统一计费口径
Core Platform

围绕“接入、路由、观测、计费”形成统一平台能力

页面重点不是堆概念，而是回答三件事：能否尽快接入、上线后是否好治理、后续扩容和切换是否省成本。

统一接入

控制台提供统一 API Key、模型目录与接口入口。开发者可以按 SDK、CLI 或第三方客户端选择最短适配路径。

  • OpenAI Chat / Responses
  • Anthropic Messages
  • Gemini Compatible Endpoints

统一治理

把接入规范、套餐权益、额度、日志、通知和模型可见性集中到控制台，便于团队内部统一口径与权限管理。

  • 用户侧控制台
  • 管理后台与运营视图
  • 套餐、订单、邀请与站点设置

统一运行期控制

平台层承接路由、限流、队列、健康度与故障切换逻辑，帮助业务方把运行期复杂度留在平台侧管理。

  • 请求排队与并发控制
  • 上游健康与失败切换
  • 日志与观测指标输出
Protocol Surfaces

按工具链选择协议入口，而不是为每个客户端重做一套接入

当前平台面向主流开发工作流提供三类入口。先匹配你的工具链，再根据目标模型与请求格式选择对应协议，能更快完成首轮联调。

OpenAI

面向 SDK、Web 应用与多数客户端

适合 OpenAI SDK、OpenAI 兼容客户端、Chat Completions 与 Responses 风格调用。通常是业务后端和多数桌面客户端的默认入口。

Anthropic

面向 Claude Code 与原生 Messages 工作流

适合 Claude Code、Anthropic SDK、偏 Messages 协议的编码 Agent 与需要保持 Anthropic 调用体验的场景。

Gemini

面向 Gemini CLI 与兼容多模态调用

适合 Gemini 风格端点或需要与 Gemini 兼容客户端对接的场景。具体模型与能力范围以控制台目录为准。

Operational Journey

从创建 Key 到运行治理，接入路径应该短且清晰

把真正的接入动作拆成清晰路径，能显著减少试错时间，也更适合团队内部标准化推进。

Step 01

创建账户并进入控制台

登录或注册后进入统一控制台，获取 API Key，并先确认模型目录、套餐与计费权益。

Step 02

按工具链选择接入方式

SDK 用户通常走 OpenAI 兼容入口，CLI 用户优先查看 Claude Code、Codex 或 Gemini 的对应配置章节。

Step 03

根据任务选择模型与路由策略

模型选择、价格和调用限制统一呈现在平台层，便于后续在不改业务入口的情况下持续调整与扩展。

Quickstart

先跑通一次请求，再进入控制台做正式治理

下面给出最常用的接入模板，目标是让你先在最短时间内验证链路可用，再按控制台口径完成正式上线配置。生产接入前，请以文档中心和控制台内实际模型目录、价格与限制说明为准。

from openai import OpenAI

client = OpenAI(
    base_url="https://api.yunjunet.cn/v1",
    api_key="sk-your-api-key"
)

resp = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "你好"}]
)

print(resp.choices[0].message.content)
import anthropic

client = anthropic.Anthropic(
    base_url="https://api.yunjunet.cn",
    api_key="sk-your-api-key"
)

message = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "你好"}]
)

print(message.content)
export ANTHROPIC_BASE_URL=https://api.yunjunet.cn
export ANTHROPIC_AUTH_TOKEN=sk-your-api-key

Governance & Scope

把能力边界写清楚，才是真正的官方与合规

页面表达必须服务于接入决策，而不是制造不必要的误解。下面这些说明会在首页、文档和控制台中保持统一口径。

能力边界

平台提供的是统一接入层与控制台能力，不直接承诺第三方模型的长期可用性、响应效果或特定输出结果。模型能力与质量由所选模型及其上游服务决定。

  • 模型目录以控制台当前可用项为准
  • 实际响应时间受模型、负载与调用方式影响
  • 兼容接口不等于所有第三方客户端功能完全一致

计费与权益

平台支持按账户权益、套餐、钱包或加油包等口径管理使用成本。用户在进行生产接入前，应优先核对控制台中的价格、配额、套餐状态和结算记录。

  • 展示页价格用于参考，不替代最终结算记录
  • 不同模型和计费模式的费用口径可能不同
  • 余额、套餐和调用限制由控制台与接口共同决定
Scenarios

适合哪些接入场景

当你同时面对多协议、多模型和多客户端时，平台层的价值会比较明确。下面这些是当前最典型的使用场景。

CLI 与编码 Agent 工作流

需要在 Claude Code、Codex、Gemini CLI 等终端工作流中统一账户、统一模型入口与统一结算口径。

AI 产品或业务服务接入

需要通过 OpenAI 或 Anthropic 兼容方式快速完成后端接入，同时保留后续调整模型和路由的空间。

团队统一治理

多个成员使用不同工具链，但希望把 API Key、使用记录、模型目录与套餐管理统一收口到同一控制台。

内部共享网关与平台化建设

希望把接入规范、计费和运行期能力沉淀为内部基础设施，而不是为每个项目单独集成一遍模型供应商。

Support & Conversion

文档、控制台、技术支持和商务合作入口保持一致

页面中出现的联系信息仅用于技术支持、商务沟通和接入咨询。若涉及价格、权益、模型目录或异常排查，请优先提供控制台信息、订单号、请求 ID 或相关截图，以便快速定位。已明确接入计划的团队，建议直接带上使用场景、预估调用规模和目标工具链，以提升沟通效率。

Next Step

已经准备开始时，按这个顺序推进最省时间

先创建 API Key

进入控制台确认账户状态、API Key 和模型目录，这是任何正式接入前的第一步。

再选接入协议

根据你的 SDK、CLI 或客户端类型，进入文档中心对应章节，不建议混用不同协议的参数结构。

最后核对结算口径
